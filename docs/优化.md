# PyKYCH 项目优化方案

> 基于对整个项目的全面代码审查，按优先级和类别整理。
> 已修复的 Bug（BUG-001~038、SEC-001）不再列入。

---

## 📐 一、架构与代码组织

### OPT-001: Jinja2 模板引擎重复初始化

- **现状**: `main.py`、`routes/admin.py`、`routes/auth.py`、`routes/search.py`、`routes/md.py` 各自独立创建了 `jinja_env` 实例，且每个文件都重复注入相同的 `globals`（`site_logo`、`site_favicon`、`site_title_func`、`site_subtitle_func`）。
- **影响**: 代码冗余（约 30+ 行重复），修改注入逻辑需改 5 个文件。
- **方案**: 提取到 `core/templates.py` 统一初始化，其他模块直接导入 `jinja_env` 或 `render()` 函数。

### OPT-002: 文章类型路由注册大量重复代码

- **现状**: `routes/admin.py` 中 5 种文章类型 × 5 个操作（new GET/POST、edit GET/POST、delete POST）= 25 个几乎相同的函数定义（L458-561），仅参数不同。
- **影响**: 新增文章类型需复制粘贴 5 个函数，维护成本高。
- **方案**: 使用循环动态注册路由，或利用 LiHiL 的路由参数特性统一为一个路由组。

### OPT-003: `main.py` 中 API 路由应独立为模块

- **现状**: `main.py` 包含了行评论 API、评分 API、当前用户 API 等大量路由定义（L327-500），导致主入口文件超过 600 行。
- **影响**: 主入口文件职责过重，不利于维护和测试。
- **方案**: 将 `/api/*` 路由提取到 `routes/api.py`，`main.py` 只做 `app.include()`。

### OPT-004: `main.py` 中硬编码文章数据

- **现状**: `/article/{slug}` 路由（L191-291）硬编码了 3 篇文章的完整 HTML 内容，这些数据与数据库中的文章系统完全独立。
- **影响**: 无法通过管理后台修改，且占用源代码空间。
- **方案**: 如果这些文章已迁移到数据库，删除此硬编码路由；如果仍需保留，移到独立的数据文件中。

### OPT-005: `legacy/` 模块清理

- **现状**: `src/pykych/legacy/` 包含 20+ 个文件的旧版代码（`_auth_legacy.py`、`article_manager.py`、`bbcode_parser.py` 等）。
- **影响**: 增加代码库体积，新开发者容易混淆。
- **方案**: 确认无其他模块引用后，删除整个 `legacy/` 目录，或移至独立的 `archive/` 仓库。

## ⚡ 二、性能优化

### OPT-006: 文章列表页 N+1 查询问题
- **现状**: `routes/md.py` L77-78 中，文章列表页对每篇文章单独调用 `get_tags_for_article()`，N 篇文章产生 N+1 次数据库查询。
- **影响**: 文章数量增长后，列表页加载变慢。
- **方案**: 新增 `get_tags_for_articles_batch()` 方法，使用 `WHERE (article_type, article_slug) IN (...)` 一次查询所有标签，再在 Python 中分组。

### OPT-007: 搜索功能 `LIKE %keyword%` 全表扫描

- **现状**: `routes/search.py` 对 5 张文章表执行 `LIKE %keyword%` UNION ALL 查询，无法利用索引。
- **影响**: 文章数量增长后搜索响应变慢（BUG-008 已记录但标记为已修复，实际代码仍为 LIKE 查询）。
- **方案**:
  1. **短期**: 为 `title` 列添加普通索引，至少加速前缀匹配；将 `content LIKE` 改为只在 `title` 中搜索，或使用 MySQL 全文索引 `FULLTEXT`。
  2. **中期**: 引入 MySQL `FULLTEXT` 索引 + `MATCH ... AGAINST` 语法。
  3. **长期**: 接入 Elasticsearch / Meilisearch。

### OPT-008: `get_all_tags_with_counts()` 复杂子查询

- **现状**: `content/tags.py` L128-153 使用包含 5 个 `EXISTS` 子查询的 `CASE WHEN` 语句，每次查询标签列表都会执行非常重的 SQL。
- **影响**: 标签数量增长后查询变慢。
- **方案**: 在 `article_tags` 删除时同步清理孤立记录（启动时 + 文章删除时级联清理），保证数据一致性后，直接 `COUNT(*)` 即可，无需 `EXISTS` 子查询。

### OPT-009: 设置文件 `set_setting()` 多次调用时的冗余写入

- **现状**: `routes/admin.py` L1205-1233 中 `update_site_settings()` 连续调用 15+ 次 `set_setting()`，每次都执行「读文件 → 改内存 → 写文件」的完整流程。
- **影响**: 一次设置保存产生 15 次文件 I/O。
- **方案**: 新增 `batch_set_settings(pairs: list[tuple[str, Any]])` 方法，一次性读取、修改所有字段、写入一次。

### OPT-010: Markdown 解析器每次请求新建实例

- **现状**: `routes/md.py` L43-64 每次请求都调用 `_create_md_parser()` 创建新的 `Markdown` 实例。虽然解决了线程安全问题（BUG-011），但实例化开销不小。
- **方案**: 使用 `asyncio.Lock()` 保护全局实例，或使用 `markdown.markdown()` 函数式 API（内部处理了线程安全）。

### OPT-011: 首页加载多次独立数据库查询

- **现状**: `main.py` L151-165 `home()` 依次执行 3 次独立查询（子站点链接、推荐文章、重要通知），每次获取/释放连接。
- **方案**: 使用 `asyncio.gather()` 并行执行无依赖的查询。

## 🔒 三、安全加固

### OPT-012: Session Cookie 缺少 `Secure` 标志
- **现状**: `main.py` L134-136 `SessionMiddleware` 未设置 `https_only=True`，在生产环境（HTTPS）下 Cookie 可能通过 HTTP 泄露。
- **方案**: 根据环境变量或配置设置 `https_only`。

### OPT-013: Secret Key 默认值不安全

- **现状**: `main.py` L129-132 当 `PYKYCH_SECRET_KEY` 未设置时使用硬编码默认值 `"pykych-secret-change-in-production"`。
- **影响**: 忘记设置环境变量时，攻击者可以伪造 Session Cookie。
- **方案**: 启动时检测默认密钥，输出醒目警告日志（类似 Django 的 `SECRET_KEY` 检查）。

### OPT-014: 文件上传缺少 MIME 类型服务端验证

- **现状**: `routes/admin.py` L928 直接使用客户端提供的 `content_type`，未做服务端验证。
- **影响**: 攻击者可上传伪装为图片的恶意文件。
- **方案**: 使用 `python-magic` 或文件头魔数检测真实 MIME 类型，与 `files.py` 的 `ALLOWED_FILE_TYPES` 白名单比对。

### OPT-015: `_save_site_asset` 使用 MD5 哈希

- **现状**: `routes/admin.py` L1252 使用 `hashlib.md5(data)` 生成文件名。
- **影响**: MD5 碰撞攻击理论上可构造同名文件覆盖。
- **方案**: 改用 `hashlib.sha256` 或 `secrets.token_hex(12)`。

### OPT-016: 头像上传使用 MD5

- **现状**: `auth/profile.py` L199 `hashlib.md5(file_data)` 同上。
- **方案**: 改用 `sha256`。

### OPT-017: 速率限制仅内存存储

- **现状**: `auth/rate_limit.py` 使用模块级字典存储（BUG-015 已记录）。
- **方案**: 抽象为接口，支持 Redis / 数据库后端。当前可作为 `RateLimitBackend` 协议，默认实现 `MemoryRateLimitBackend`，预留 `RedisRateLimitBackend`。

## 🧹 四、代码质量

### OPT-018: 大量 `from starlette.responses import ...` 延迟导入

- **现状**: `main.py` 中多处函数内部 `from starlette.responses import JSONResponse`（L367, L373, L380 等），重复导入。
- **方案**: 在文件顶部统一导入 `JSONResponse`。

### OPT-019: `admin.py` 中权限检查返回的仪表盘数据冗余

- **现状**: `_csrf_error()` 和 `_require_owner()` 中构造 `admin_dashboard.html` 时传入了大量 `0`、`[]` 占位数据（L93-100, L131-137），共约 20 个参数。
- **方案**: 提取为 `_empty_dashboard_context()` 辅助函数，或让模板对缺失变量使用默认值。

### OPT-020: `delete_user()` 中 `tag_id` 子查询只覆盖 `articles` 表

- **现状**: `auth/user.py` L294-298 删除 `article_tags` 时，子查询只查 `articles` 表：
  但用户可能还有 `pages`、`html_pages`、`bbcode_pages`、`typst_pages` 的标签关联。
- **方案**: 扩展为覆盖所有 5 张文章表的 UNION 子查询，或在删除文章前先行清理标签。

### OPT-021: `get_user_by_username()` 使用 try/except 检测列是否存在

- **现状**: `auth/user.py` L51-63 用 try/except 包裹两条 SELECT 语句来检测 `avatar` 列是否存在。
- **影响**: 每次查询用户时，如果 `avatar` 列不存在，会触发异常再回退，异常处理开销较大。
- **方案**: 启动时一次性检测列是否存在并缓存结果（`ensure_profile_columns()` 已有类似逻辑），运行时使用确定的字段列表。

### OPT-022: `_generate_snippet()` 每次调用都 `import re`

- **现状**: `routes/search.py` L159 在函数内部 `import re`。
- **方案**: 移至文件顶部导入。

### OPT-023: `_save_site_asset` 函数内部重复导入

- **现状**: `routes/admin.py` L1240-1241 每次调用都 `import os, hashlib` 和 `from pathlib import Path`。
- **方案**: 移至文件顶部。

### OPT-024: `upload_theme()` 函数内部大量重复导入

- **现状**: `routes/admin.py` L1303 `import zipfile, io, tempfile, shutil`、L1342 `import yaml`、L1349 `import re` 均在函数内部导入。
- **方案**: 移至文件顶部。

## 🗄️ 五、数据库优化

### OPT-025: 五张文章表结构完全相同，可考虑合并

- **现状**: `articles`、`pages`、`html_pages`、`bbcode_pages`、`typst_pages` 五张表结构完全相同（`id, slug, title, content, author_id, created_at, updated_at`），但分散存储。
- **影响**: 所有跨类型查询都需要 `UNION ALL`，搜索、统计、推荐等操作复杂度翻倍。
- **方案**: 合并为统一的 `articles` 表，增加 `article_type ENUM` 列。这是最大的架构改动，建议作为长期目标。
- **替代方案**: 保持分表，但为跨表查询建立视图 `VW_ALL_ARTICLES`。

### OPT-026: 缺少数据库索引

- **现状**: 部分表缺少常用查询的索引：
  - `comments` 表缺少 `(article_type, article_slug)` 组合索引（已有，但 `author_name` 无索引）
  - `ratings` 表 `author_name` 无索引（`delete_user()` 按 `author_name` 删除时需要全表扫描）
  - `static_files.uploaded_by` 无索引
- **方案**: 为频繁用于 `WHERE` 条件的列添加索引。

### OPT-027: `delete_article()` 未使用事务 ✅

- **已完成**: 代码已使用 `await conn.begin()` / `commit()` / `rollback()` 包裹 5 条 DELETE 语句，确保原子性。

### OPT-028: `featured_articles` 排序交换使用两条 UPDATE ✅

- **已完成**: 排序交换操作使用 `conn.begin()` / `commit()` / `rollback()` 事务包裹，确保并发安全。

## 🧪 六、测试与可维护性

### OPT-029: 项目缺少自动化测试

- **现状**: 未发现 `tests/` 目录或任何测试文件。
- **方案**: 添加核心模块的单元测试，优先覆盖：
  1. `auth/password.py` — 密码哈希/验证
  2. `core/settings.py` — 设置读写
  3. `content/articles.py` — 文章 CRUD
  4. `auth/session.py` — CSRF Token 生成/验证

### OPT-030: 缺少类型标注

- **现状**: 部分函数缺少返回类型标注或参数类型标注（如 `main.py` 中的路由函数、`admin.py` 中的辅助函数）。
- **方案**: 逐步添加完整的类型标注，配合 `mypy` 检查。

### OPT-031: 缺少 API 文档

- **现状**: 项目有 LiHiL 框架支持自动生成 OpenAPI 文档，但路由中缺少 `summary`、`description` 等元数据。
- **方案**: 为每个路由添加文档字符串和元数据。

## 🚀 七、部署与运维

### OPT-032: Docker 构建未利用缓存层

- **现状**: 未检查 `Dockerfile`，但 `pyproject.toml` 的依赖安装可能未分离缓存层。
- **方案**: 确保 `Dockerfile` 先复制 `pyproject.toml` 并 `pip install`，再复制源代码。

### OPT-033: 缺少健康检查深度

- **现状**: `/health` 端点只返回静态信息，不检查数据库连接状态。
- **方案**: 添加数据库连接检查。

### OPT-034: 缺少结构化日志配置

- **现状**: 各模块使用 `logging.getLogger(__name__)` 但未配置统一的日志格式、级别和输出目标。
- **方案**: 在 `main.py` 启动时配置 `logging.basicConfig()` 或使用 `dictConfig()`，支持 JSON 格式输出（方便日志收集系统解析）。

### OPT-035: 静态文件未设置缓存头
- **现状**: `serve_upload`、`serve_static_img`、`serve_avatar` 等使用 `FileResponse` 但未设置 `Cache-Control` 头。
- **影响**: 浏览器每次访问都重新请求静态资源。
- **方案**: 为静态文件设置合理的缓存头。

## 🗄️ 五、数据库优化

### OPT-025: 五张文章表结构完全相同，可考虑合并 ❌

- **现状**: `articles`、`pages`、`html_pages`、`bbcode_pages`、`typst_pages` 五张表结构完全相同，但分散存储。
- **影响**: 所有跨类型查询都需要 `UNION ALL`，搜索、统计、推荐等操作复杂度翻倍。
- **方案**: 合并为统一的 `articles` 表，增加 `article_type ENUM` 列。这是最大的架构改动，建议作为长期目标。
- **替代方案**: 保持分表，但为跨表查询建立视图 `VW_ALL_ARTICLES`。

### OPT-026: 缺少数据库索引 ❌

- **现状**: 部分表缺少常用查询的索引：
  - `ratings` 表 `author_name` 无索引（`delete_user()` 按 `author_name` 删除时需要全表扫描）
  - `static_files.uploaded_by` 无索引
- **方案**: 为频繁用于 `WHERE` 条件的列添加索引。

### OPT-027: `delete_article()` 未使用事务 ✅

- **已完成**: `content/articles.py` L281-314 已使用 `async with conn.begin()` / `commit()` / `rollback()` 包裹所有删除操作。

### OPT-028: `featured_articles` 排序交换使用两条 UPDATE ❌

- **现状**: `core/site_settings.py` L200-208 交换 `sort_order` 使用两条独立的 UPDATE 语句，未使用事务。
- **影响**: 并发操作时可能出现排序冲突。
- **方案**: 使用事务包裹，或使用 `CASE WHEN` 在单条 SQL 中完成交换。
